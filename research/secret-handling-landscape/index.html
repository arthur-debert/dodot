    Secret Handling in Dotfiles Managers: A Landscape Survey¶

    A comparative look at how the mainstream dotfiles / personal configuration tools handle sensitive data — API tokens, SSH keys, credentials, personal identifiers — and the primitives (age, SOPS, git-crypt, password-store, 1Password, etc.) they compose with. The goal is to inform what dodot should offer.

    1. Tool-by-tool¶

    Each entry uses the same mini-template: identity & philosophy, secret mechanism, UX flow (add / bootstrap / rotate / share), and trade-offs.

    1.1 chezmoi¶

    • Identity. Go binary, template-renderer model (not a symlink farm). The source state is a directory whose files are copied/rendered/templated into $HOME on chezmoi apply. Mature and broadly adopted; arguably the reference implementation for dotfiles-with-secrets. Docs: https://www.chezmoi.io.
    • Design philosophy. Declarative destination state expressed through templates (Go text/template + sprig), with first-class template functions for almost every secret backend in existence.
    • Secret mechanism. Two orthogonal strategies, which can be combined:
    • Whole-file encryption at rest. Supports age, GPG, git-crypt, and transcrypt. Files are stored ASCII-armored in the source directory with an encrypted_ filename prefix. See https://www.chezmoi.io/user-guide/encryption/ and https://www.chezmoi.io/user-guide/encryption/age/.
    • Reference-from-vault templates. Template functions pull secrets at apply time from: 1Password, Bitwarden, Dashlane, Keeper, LastPass, KeePassXC, Proton Pass, pass, passhole, gopass, AWS Secrets Manager, Azure Key Vault, HashiCorp Vault, Doppler, ejson, macOS Keychain, Windows Credential Manager. See https://www.chezmoi.io/user-guide/password-managers/.
    • UX — add a new secret.
    • Whole file: chezmoi add --encrypt ~/.ssh/id_rsa (transparent re-encrypt on chezmoi edit).
    • Templated value: write .zshrc.tmpl containing e.g. export GH_TOKEN={{ onepasswordRead "op://Personal/GitHub/token" }}.
    • UX — bootstrap on a fresh machine. Install chezmoi, then chezmoi init --apply <git-url>. The machine must already have either the age/GPG key present (for encrypted files) or be signed in to the chosen vault (for template-rendered secrets). Bootstrap scripts (run_once_before_*.sh) are commonly used to install and sign in to the password-manager CLI before secrets are referenced.
    • UX — rotate. For encrypted files: chezmoi edit just works. For vault references: rotate in the vault, then chezmoi apply.
    • UX — share public. Vault-reference templates are the killer feature here: the repo contains only pointers (op://...), so it is safe to publish.
    • Trade-offs.
    • Right: broadest backend support in the space by a wide margin; single file can mix encrypted blobs and vault references; declarative; cross-platform (Linux/macOS/Windows).
    • Wrong: template complexity scales rapidly (Go template syntax, not beginner-friendly); vault reference approach requires CLI to be installed and authenticated on every machine, which forks the bootstrap story.

    1.2 yadm¶

    • Identity. A thin wrapper over git that treats $HOME directly as the working tree. Bash. Docs: https://yadm.io.
    • Design philosophy. Imperative, git-native. Whatever git does, yadm does — plus a few conventions (alternates, bootstrap hook, encrypt archive).
    • Secret mechanism. Three options, documented side-by-side at https://yadm.io/docs/encryption:
    • Built-in yadm encrypt / yadm decrypt — produces a single GPG- or OpenSSL-encrypted tarball ($HOME/.local/share/yadm/archive) containing everything that matches $HOME/.config/yadm/encrypt patterns. The archive is committed; the plaintext files are git-ignored.
    • git-crypt — yadm git-crypt ... passes through; transparent per-file encryption via .gitattributes filters.
    • transcrypt — same passthrough pattern; symmetric passphrase.
    • UX — add a new secret.
    • Built-in: append /.ssh/*.key to .config/yadm/encrypt, then yadm encrypt, then yadm add .local/share/yadm/archive and commit.
    • git-crypt: yadm git-crypt init, configure .gitattributes, commit.
    • UX — bootstrap. yadm clone <url> followed by yadm decrypt (with the passphrase / GPG key) or yadm git-crypt unlock. Yadm also has a bootstrap hook that runs automatically after clone.
    • UX — rotate. Built-in archive: re-run yadm encrypt, commit new archive. git-crypt: normal git-crypt rekey flow (awkward).
    • UX — share public. Repo is commit-safe with any of the three approaches, though yadm's own docs still recommend a private repo as defence-in-depth.
    • Trade-offs.
    • Right: encryption is native, not bolted on; archive approach is straightforward to understand (one tarball, one passphrase); easy upgrade path from a plain git-managed $HOME.
    • Wrong: archive approach loses per-file history (the archive is opaque to diff); built-in encrypt is GPG/OpenSSL only — no age; no vault integration.

    1.3 dotbot¶

    • Identity. Python. Symlink-farm + imperative shell hooks driven by a YAML plan. https://github.com/anishathalye/dotbot.
    • Design philosophy. Deliberately minimal and unopinionated. The tagline is roughly "dotbot does less than you think because git does more than you think." Idempotent plan execution is the core.
    • Secret mechanism. None natively. Secrets are handled by third-party plugins that extend the plan schema:
    • dotbot-age — age encryption of selected files.
    • dotbot-gitcrypt — invokes git-crypt as a plan step. Users frequently just put their encrypted store alongside the dotbot repo and reference it from a shell: step.
    • UX. A typical plan adds a git-crypt: or age: entry; bootstrap is ./install, which runs the plan.
    • Trade-offs.
    • Right: extremely composable; no opinion means no wrong opinion.
    • Wrong: every user re-invents the same bootstrap glue; no standard place for secrets, which means no shared mental model across dotbot repos.

    1.4 rcm¶

    • Identity. C/shell, from thoughtbot. Symlink farm with tags and host overrides. https://github.com/thoughtbot/rcm.
    • Secret mechanism. None. rcup, rcdn, mkrc, lsrc are symlink operations only. Users handle secrets entirely out-of-band — typically a separate encrypted store (pass, git-crypt on a private repo) that they source from their rc files.
    • Trade-offs. Simple and stable; offloads the entire problem. No guidance, no answers.

    1.5 GNU Stow¶

    • Identity. Perl. The oldest and purest symlink-farm manager. https://www.gnu.org/software/stow/.
    • Secret mechanism. None and by design. Stow only creates symlinks.
    • Typical composition. Users keep secrets either (a) in a private repo stowed alongside the public one, or (b) referenced via pass / 1Password CLI invoked from their shell rc.
    • Trade-offs. Zero opinion, zero help. Whatever you layer on top is your problem.

    1.6 home-manager / nix-darwin (via sops-nix or agenix)¶

    These are not dotfile managers in the classic sense — they are Nix modules that render a user's environment — but they are the de-facto "declarative dotfiles" for the Nix community, and both mainstream secret modules are worth covering.

    1.6.1 sops-nix¶

    • Repo. https://github.com/Mic92/sops-nix. Builds on Mozilla/getsops SOPS.
    • Mechanism. Secrets are stored as SOPS-encrypted files (YAML/JSON/INI/env) in the flake/repo. Encryption backends: age, GPG, AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault. The module decrypts at NixOS activation time (nixos-rebuild switch) using the target machine's SSH host key converted to an age identity, writing plaintext into /run/secrets/<name> with declared owner, group, and permissions. The Nix store itself stays clean.
    • UX — add. Edit .sops.yaml to list recipients (personal age keys + the machine's age-converted SSH host key), then sops secrets.yaml — SOPS opens $EDITOR with plaintext and re-encrypts on save. Commit.
    • UX — bootstrap. The host's SSH host key must already exist (it normally does, since sshd generates it). Personal age key must be present for editing.
    • Trade-offs. Right: only leaf values encrypted — diffs are meaningful; many KMS backends; atomic activation. Wrong: non-trivial setup (.sops.yaml, key conversion); assumes Nix.

    1.6.2 agenix¶

    • Repo. https://github.com/ryantm/agenix. A deliberately simpler alternative to sops-nix.
    • Mechanism. Whole-file age encryption only. Recipients are SSH public keys (local, ssh-keyscan'd from running hosts, or pulled from GitHub). Encrypted .age files ship into the Nix store; the module decrypts them at activation using the host's SSH private key and mounts plaintext at /run/agenix/<name>.
    • UX — add. agenix -e secret1.age — decrypts (if present), opens $EDITOR, re-encrypts.
    • Trade-offs. Right: dead-simple; no GPG; reuses existing SSH keys. Wrong: whole-file only (no leaf-value diffs); no MACs on age files, so a hostile committer could corrupt secret contents without detection (sops-nix's explicit pitch for itself).

    1.7 git-crypt¶

    • Repo. https://github.com/AGWA/git-crypt. C++.
    • Mechanism. A git clean/smudge filter driven by .gitattributes. Files marked filter=git-crypt diff=git-crypt are AES-256-CTR encrypted on git add, decrypted on checkout, transparently. Keys are shared via GPG (git-crypt add-gpg-user <key-id>) or exported as a symmetric key (git-crypt export-key).
    • UX — add. git-crypt init; edit .gitattributes to mark files (secrets/* filter=git-crypt diff=git-crypt); commit normally.
    • UX — bootstrap. On a new machine: clone, then git-crypt unlock /path/to/symkey or git-crypt unlock with a GPG key already in the keyring.
    • UX — rotate. No real rotation. Adding a new collaborator re-encrypts forward, but the old key stays valid for history.
    • Trade-offs. Right: transparent to git workflow once configured; semantically secure under deterministic chosen-plaintext. Wrong: key rotation is basically "start a new repo"; breaks some git GUIs; filenames and commit messages are not protected.

    1.8 transcrypt¶

    • Repo. https://github.com/elasticdog/transcrypt. Bash, OpenSSL-backed.
    • Mechanism. Same clean/smudge-filter approach as git-crypt, but symmetric passphrase only. Each file gets a deterministic per-file salt derived from HMAC-SHA256(filename, password), so identical plaintexts encrypt identically and the filter stays stable across commits.
    • UX. transcrypt to init; transcrypt --add path to mark; clone and transcrypt -c aes-256-cbc -p '<passphrase>' to unlock.
    • Trade-offs. Right: pure Bash, no compile; one passphrase, easy to teach. Wrong: symmetric only — can't revoke a single collaborator; the passphrase is a single point of failure; no multi-recipient model.

    1.9 BlackBox¶

    • Repo. https://github.com/StackExchange/blackbox. Archived November 2025 with an explicit "DO NOT USE" notice. Included here for completeness because it shows up in older dotfile write-ups.
    • Mechanism. Per-file GPG, multi-recipient. Admins are added by GPG key ID; any admin can decrypt; blackbox_addadmin re-encrypts all files for the new recipient.
    • Verdict. Abandoned; do not adopt. The design (GPG multi-recipient whole-file) is essentially superseded by SOPS + age.

    1.10 Password-store (pass) and gopass¶

    These are not dotfile managers — they are password managers — but they show up constantly as the source that dotfile managers pull from.

    • pass. https://www.passwordstore.org/. One GPG-encrypted file per secret in ~/.password-store, organized by directory. Backed by git for sync. Commands: pass init <gpg-id>, pass insert foo/bar, pass -c foo/bar, pass git push.
    • gopass. https://github.com/gopasspw/gopass. Drop-in compatible; adds age backend, multiple stores, team workflows, Windows support, fuzzy-search, browser integration.
    • Composition with dotfiles. The dotfile repo contains neither secret nor encrypted blob — only a shell snippet like export GH_TOKEN="$(pass show tokens/github)" in .zshrc. Secrets live in a separate (possibly also versioned) password-store.

    1.11 SOPS and age as primitives¶

    • age. https://github.com/FiloSottile/age. Minimal file-encryption tool with age1... keys and SSH-key compatibility. "No config options" is its motto. Has become the default for new secret tooling (agenix, sops-nix prefers it, chezmoi recommends it).
    • SOPS. https://github.com/getsops/sops. Mozilla-originated, now a CNCF sandbox project. Operates on structured files (YAML/JSON/INI/env) and encrypts only leaf values, leaving keys and structure in cleartext — so diffs are meaningful. Backends: age, GPG, AWS KMS, GCP KMS, Azure Key Vault, Vault, Huawei KMS. Usage: sops edit secrets.yaml, sops -d, sops encrypt -i.
    • Who uses them in the dotfiles space. chezmoi (age, GPG directly — SOPS not natively, but trivial to call via a template output function); dotbot-age; agenix (age only); sops-nix (SOPS). Yadm has not adopted either. Stock git-crypt and transcrypt predate age and use their own primitives.

    1.12 1Password / Bitwarden CLI integration patterns¶

    Pattern used by chezmoi, and available to any dotfile manager that can shell out at apply time:

    • User installs op (1Password) or bw (Bitwarden) on the machine and signs in once (often with OS biometrics as the unlock).
    • The dotfile repo contains templates like {{ onepasswordRead "op://Personal/github/token" }} or {{ (bitwarden "item" "example.com").login.password }}.
    • At apply time, the dotfile tool invokes the CLI, gets a fresh value, and writes the rendered file to disk with 0600 perms.
    • Consequence: the repo is fully public-safe; the machine must have a live session with the vault at apply time.

    2. Design-dimension comparison¶

    2.1 Encryption-at-rest vs reference-only vs blob-at-apply¶

    Approach Examples What's in the repo Decryption requires
    Transparent encryption in git git-crypt, transcrypt, blackbox Ciphertext, filter config in .gitattributes Key available locally before checkout
    Encrypted blobs (whole files) chezmoi encrypted_*, yadm archive, agenix, dotbot-age Ciphertext files with a marker Key available before apply
    Encrypted leaf values SOPS (and sops-nix) Mixed plaintext structure + encrypted values Key available before apply
    Reference-only / vault pull chezmoi + 1Password/Bitwarden/pass/Vault, gopass-integrated shell configs Placeholders/pointers Authenticated vault session at apply time

    2.2 Bootstrap cost on a fresh machine¶

    From cheapest to most expensive:

    1. transcrypt / yadm-archive — one passphrase, typed once. No prior state.
    2. git-crypt with symmetric key — one file, copied in out-of-band.
    3. age / agenix / sops-nix — one key file (or SSH key that's already generated anyway). Portable.
    4. GPG-based anything — requires a GPG identity, keyring, often a hardware token or passphrase. High friction on a brand-new machine.
    5. Vault-reference (1Password / Bitwarden / Vault) — requires install + sign-in + network. Highest cost on day one, lowest cost thereafter because rotation doesn't touch the repo at all.

    2.3 Trust root¶

    Trust root Tools
    Passphrase in user's head transcrypt, yadm (OpenSSL/GPG-symmetric), age passphrase mode
    Local keyfile age, agenix (SSH priv key), chezmoi-age
    GPG private key (possibly on hardware token) git-crypt, pass, gopass, blackbox, chezmoi-gpg, yadm-GPG
    Machine SSH host key sops-nix, agenix (for activation-time decryption)
    Cloud KMS SOPS backends, chezmoi AWS/GCP/Azure/Vault functions
    Third-party vault session chezmoi with 1Password / Bitwarden / LastPass / Dashlane / Keeper / pass / Vault / Doppler
    OS keyring chezmoi with macOS Keychain / Windows Credential Manager

    2.4 Whole-file vs leaf-value secrets¶

    • Whole-file: git-crypt, transcrypt, yadm archive, agenix, chezmoi encrypted_*, dotbot-age, blackbox. Simplest mental model; diffs are opaque.
    • Leaf-value interpolation: chezmoi templates (with either vault lookup or embedded encrypted maps), SOPS / sops-nix. Diff-friendly, review-friendly, but more moving parts.

    Most dotfile secrets (SSH keys, GPG keys, age keys, certificates) are binary-ish and want whole-file encryption. Most config secrets (API tokens, .npmrc, .aws/credentials) want leaf-value interpolation. Good designs accommodate both.

    2.5 Team / multi-user vs single-user multi-machine¶

    • Single-user multi-machine (the common dotfiles case) — a symmetric passphrase, a single age keyfile, or a vault with biometric unlock on each machine are all adequate. transcrypt, age, 1Password-style integrations shine here.
    • Team / multi-user — requires multi-recipient encryption (git-crypt GPG mode, SOPS with multiple recipients, blackbox, agenix with multiple SSH keys) or a shared vault. Symmetric schemes (transcrypt) break down because passphrase rotation means a group-wide re-keying event.

    3. Synthesis table¶

    Tool Mechanism Bootstrap cost Biggest strength Biggest weakness
    chezmoi (age/GPG) encrypted whole-file blobs keyfile + chezmoi binary transparent edit; same repo mixes encrypted and plaintext age keyfile must reach the machine out-of-band
    chezmoi (1Password/Bitwarden/Vault) reference-only templates CLI install + signed-in session repo is fully public-safe; rotation free needs live vault at every apply
    yadm (built-in) GPG/OpenSSL tarball archive passphrase or GPG key one command, one archive opaque diffs, whole-archive rotation
    yadm + git-crypt transparent git filter GPG key or symkey file git-native git-crypt's own rotation / GUI warts
    dotbot + plugins whatever the plugin does plugin-dependent flexible no standard — every repo differs
    rcm — n/a simple no answer for secrets at all
    GNU Stow — n/a pure symlinks no answer
    sops-nix SOPS leaf-value encryption SSH host key + personal age key diff-friendly; many KMS backends Nix-only; non-trivial setup
    agenix age whole-file in Nix store SSH host key very simple, SSH-key-native whole-file only; no MAC
    git-crypt (standalone) git clean/smudge AES GPG key or symkey transparent git workflow no real rotation story
    transcrypt (standalone) git filter, symmetric OpenSSL one passphrase Bash-only, zero install symmetric = no revoke
    blackbox per-file GPG multi-recipient GPG keyring multi-admin sharing abandoned
    pass / gopass GPG (and age, gopass) per-secret file GPG or age key canonical Unix; composes with anything separate from dotfiles; adds a tool to learn
    SOPS + age leaf-value encryption, many backends key or KMS access diff-friendly; KMS-friendly not a dotfile tool — you wire it in yourself

    4. What the good designs have in common¶

    1. They separate "secret lives here" from "dotfile points at it" whenever possible. The repo either contains encrypted blobs with a published decryption story, or contains only references to a vault. It does not contain plaintext that "you just shouldn't commit." The best tools make the plaintext-in-repo state impossible, not discouraged.
    2. Transparent edit beats explicit encrypt/decrypt. chezmoi edit, agenix -e, sops secrets.yaml, blackbox_edit, and git-crypt's clean/smudge filter all implement the same pattern: the user interacts with plaintext in their $EDITOR, and the tool handles cipher/plaintext transitions. Users do not want to remember to re-encrypt.
    3. Age is winning over GPG for new designs. agenix, sops-nix's preferred mode, chezmoi's recommended default, dotbot-age, gopass's non-default backend — every greenfield tool is picking age. GPG remains only in ecosystems where it's already installed for other reasons (pass, yadm). The lesson: a dotfiles tool that standardises on age as its lowest-friction default and supports "bring your own" (GPG, vault) for people who have them is probably on the right side of this trend.
    4. The 1Password/Bitwarden-CLI pattern is quietly the best UX for single-user multi-machine. Once signed in (often via biometrics), the user never touches a key, a passphrase, or a rotation ceremony. The cost — requiring a vault session at apply time — is usually acceptable. This is what makes chezmoi's template-function approach powerful and worth imitating.
    5. Leaf-value encryption is better for config-style secrets; whole-file for key material. The tools that cover both (chezmoi, SOPS-based stacks) give users the right ergonomics for each case. Tools that only do one (agenix whole-file, yadm archive whole-file) force awkwardness on the other.
    6. Bootstrap is the failure mode. Every secret scheme works fine once it's set up. What distinguishes good designs is how painful the "new laptop, no keys yet" situation is. The best answers in the field are: (a) a single age keyfile you copy from another machine over a USB stick or AirDrop; (b) a vault biometric unlock; (c) an SSH host key that already exists. GPG-based answers are consistently the worst.

    5. Implications for dodot¶

    These are observations, not a design spec:

    • Default should be age. It's the low-friction modern primitive and aligns with where the ecosystem is heading.
    • Template/placeholder story matters even if encryption-at-rest is the headline feature. Users will want .npmrc-style leaf-value secrets.
    • Plan for the "bring your own vault" case. 1Password/Bitwarden CLI calls from within rendered files are high-value and relatively simple to support (shell out, capture stdout, write 0600).
    • Do not invent key management. Reuse SSH keys and/or age keys. Do not ship a bespoke keystore.
    • Decide early whether dodot wants to be opinionated (one blessed path, like agenix) or comprehensive (every backend, like chezmoi). The comprehensive path is a long tail of integration work; the opinionated path loses users with existing vault investments. Hybrid: pick one blessed local primitive (age) plus one blessed vault-reference pattern (shell out to a configurable CLI), and stop.
    • Make plaintext-in-repo impossible, not just unrecommended. A dodot add --encrypt workflow that actively refuses to commit plaintext for anything it has seen encrypted once is the kind of ergonomic guard rail the existing tools mostly lack.
